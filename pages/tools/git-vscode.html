Create Issue and Branch

Assume there is an issue to work on. Go to the “Issues” tab to create a new issue: “My Task”.


Create a new local branch “feature/1-My-Task”, or use another naming convention to name your branches. Click on the “main” branch in the bottom left corner and select “Create new branch”.


Type branch name and click Enter.


In the bottom left corner note that a new branch is selected.

Imitate Chages Made by Another Developer

Assume that another developer worked on their branch and pushed changes to the “main” branch. Let’s imitate this situation. Go to the GitHub repository and edit “file1.txt” directly in the “main” branch.

Commit Changes

Note, that your code was written on lines 4 and 5. And now these lines are occupied by another code (what is known as a “Git conflict”). Return back to VS Code. Assume you don’t know about these changes. Simply commit a new branch with your changes.


Copy-Paste issue description to the commit description.



Outline


Create Pull Request and Observe Merge Conflict

Go to the GitHub repository, you will see a popup “Compare & pull request”.


If there is no auto popup, then go to the “Pull requests” tab and select “New pull request”.


Select your branch to merge into the “main” branch.


There is a message that GitHub “Can’t automatically merge” “feature/1-My-Task” branch into the “main” branch.


The reason behind this is that you created your branch from a “main” branch state, which is not relevant any more (now it is in the “past”, since another developer updated “main” branch). To resolve this, it's required firstly to update “feature/1-My-Task” branch, and after this push our commits. When another developer was working on their task, perhaps they didn’t know about our work on our branch. And that’s why new code from the “main” branch may affect the same files and lines of code that we worked on. Synchronization of these changes is known as “Git conflict resolving”.

Outline


Merge Updates from Main-Branch to Your Feature-Branch.
Resolve Merge Conflict

Return back to VS Code. Checkout to the “origin/main” branch (in the bottom left corner).


There is an initial state of the “main” branch.


Use circle with arrows to get updates from the “origin/main” branch.


Now, we have an updated state of the “main” branch including changes made by another developer.


However, there is still no connection between these updates and our branch “feature/1-My-Task”. To fix this we need to merge the “origin/main” branch to “feature/1-My-Task”. Checkout to “origin/feature/1-My-Task”.


Go to Source Control tab (left side panel) > Three dots > Branch > Merge Branch. Select a branch to merge from. A branch to merge from is “origin/main” (it will be merged with its updates to the “feature/3-My-Task” branch).



As expected, there is a Git merge conflict.


Click on the “file1.txt” in the explorer to open the Resolve conflict manager.


There are three columns: (1) Incoming from “main” branch, (2) initial state of “main” branch, and (3) our code. Below there is a Result column that will contain the final version of our code including updated code from the “main” branch.

The point is that another developer and we worked on the same file and same lines of code, so the Git doesn't know which code has a priority to stay. We have to explicitly command how a final version of our code should look like. The simplest option is to select Three dots > Accept All Changes from Left.



Similarly, Three dots > Accept All Changes from Right.



However, it might be not what we want. So, we can type what we need in the Result section directly.


Click on a “Complete Merge” button. Git conflict is resolved. Now, we can commit our code (including changes merged from the “main”branch).



Outline


Create New Pull Request

Return back to a GitHub repository. Click “Compare & pull request”.


Now, we see that GitHub is able to merge the “feature/1-My-Task” branch into the “main” branch.


Click “Create pull request”. Then “Merge pull request” or “Squash and merge” (if you don’t know what to choose, then choose “Merge pull request”).


Now, “file1.txt” in the “main” branch looks exactly as we defined during merge conflict resolution.


Outline


Finish Work in VS Code

Return back to VS Code. Now you are on your “feature/1-My-Task” branch. Since your work is completed, checkout to the “origin/main” branch.


As we can see, “file1.txt” has code before merging.


Click on a circle with arrows to get updates from the "origin/main” branch, including your previously pushed code.


Now, we have a current state of the “main” branch on our local machine.
